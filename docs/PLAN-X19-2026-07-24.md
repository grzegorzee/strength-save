## ROOT CAUSE (zweryfikowany w kodzie 2026-07-24)

**Dwa formaty id dni:**
- Start cyklu (`startCycleWithPlan`, `completeOnboardingPlan` w `cycle-actions.ts:76,118`) woła `assignCycleDayIds(days, startDate)` (`plan-cycle-utils.ts:77-78`) → dni dostają `${startDate}-dN` i TE SAME id trafiają do planu (`savePlan(uniqueDays)`). Po starcie cyklu plan i cykl są spójne.
- Format `day-N` żyje w: domyślnym planie (`trainingPlan` w `data/trainingPlan.ts`), builderze i szablonach.

**Mechanizm rozjazdu:** `saveTrainingPlanWithRevision` (`training-plan-save.ts:67-70`) przy `syncActiveCycle !== false` woła `buildActiveCyclePlanPatch(newPlan, ...)` (`plan-cycle-utils.ts:132-140`), który podmienia `cycle.days` 1:1 na dni planu — z ich AKTUALNYMI id, bez żadnego mapowania.

**Zweryfikowane wektory dryfu (plan z `day-N` przy aktywnym cyklu z `${startDate}-dN`):**
1. `resetToDefault()` w PlanEditor → `savePlan(defaultPlan)` (id `day-N`) + domyślny sync → dni aktywnego cyklu zmieniają id na `day-N`, podczas gdy dokumenty `workouts` i drafty z tego cyklu noszą `dayId = ${startDate}-dN`.
2. `addPlanDay` (`plan-day-edit.ts:13-18,30`) ZAWSZE nadaje `day-N` (pierwszy wolny N), nawet gdy pozostałe dni mają format cyklu → dodanie dnia w PlanEditor w trakcie cyklu wstrzykuje do cyklu dzień w obcym formacie (mieszanka formatów w jednym cyklu).
3. Zapis planu z buildera/szablonu podczas aktywnego cyklu (dni szablonu mają własne id).

**Skutki (do potwierdzenia zakresu w Z150):** nawigacja Dashboardu używa `todayTraining.dayId` / `item.dayId` (`Dashboard.tsx:615,1013`); drafty treningu kluczowane `(uid, dayId, date)`; `findWorkoutForRoute` ma `allowDateFallback` (łagodzi część przypadków); prefiks id ćwiczeń zależy od formatu id dnia (`exercisePrefixForDay`, `plan-cycle-utils.ts:82-85`). Zmiana id dnia w trakcie cyklu = ryzyko: druga sesja na ten sam dzień kalendarzowy, draft nieodnajdywalny po edycji planu, mismatch prefiksu nowych id ćwiczeń.

## USTALENIA Z150 (2026-07-24, read-only)

### 1. (a) Skąd Dashboard bierze `dayId`

- Karta "dzisiejszy trening": `todayTraining` (`Dashboard.tsx:322-362`) bierze dzień z `getScheduledTrainingForDate(trainingPlan, today)`, gdzie `trainingPlan` pochodzi z `useTrainingPlan` (kolekcja `training_plans`) — czyli **dni PLANU, nie cyklu**. Nawigacja: `/workout/${todayTraining.dayId}?date=...&autostart=true` (`Dashboard.tsx:615`).
- Karty tygodnia: `thisWeek` też z planu, `item.dayId = day.id` (`Dashboard.tsx:976`), nawigacja `Dashboard.tsx:1011-1013`.
- Wniosek: cykl NIE jest źródłem `dayId` nawigacji. Po zapisie planu z podmienionymi id wszystkie linki natychmiast wskazują NOWE id, a historia/drafty zostają przy starych.
- Karta "kontynuuj trening": warunek `localDraft.dayId === todayTraining.dayId` (`Dashboard.tsx:587-588`) — po podmianie id żywy draft przestaje być kontynuowalny z karty i guzik przechodzi w autostart NOWEJ sesji.

### 2. (b) Żywy draft `(uid, dayId, date)` po zmianie id dnia

- IDB store kluczowany `(userId, sessionId)` (`workout-draft-db.ts:454`); adopcja draftu w WorkoutDay wymaga `draft.dayId === dayId && draft.date === targetDate` (`WorkoutDay.tsx:282-286`).
- Powrót ścieżką auto-resume / "kontynuuj": URL budowany ze STAREGO `draft.dayId` (`workout-resume.ts`: `continuableDraftTarget`, `shouldResumeWorkoutDraft`). Wtedy `baseDay = trainingPlan.find(d => d.id === dayId)` nie znajduje dnia (`WorkoutDay.tsx:277`): draft z treścią → `buildDayFromDraft(undefined, draft)` renderuje dzień TYLKO z dotkniętych ćwiczeń (`workout-day-view.ts:30`, planExercises=[]); draft bez treści → ekran `workout.dayNotFound` (`WorkoutDay.tsx:1935-1939`).
- Powrót przez kartę Dashboardu (nowe id): `baseDay` jest, draft nie pasuje → autostart NOWEJ sesji na ten sam dzień kalendarzowy; stary draft osierocony w IDB (serie niezsynchronizowane znikają z pola widzenia usera).

### 3. (c) `findWorkoutForRoute` + `allowDateFallback` (`workout-lookup.ts:50-57`)

- Historia (`date < today`): fallback po dacie zwraca najlepszy UKOŃCZONY trening z tej daty niezależnie od `dayId` → wejście w historię po zmianie formatu DZIAŁA (łagodzi skutek).
- Dziś (`date >= today` przy podanym `dayId`): fallback celowo zablokowany → ukończony dziś trening pod starym id jest niewidoczny dla karty dnia z nowym id → karta pokazuje dzień jako niezrobiony i autostartuje DRUGĄ sesję. Karta zbiorcza "completed today" (`Dashboard.tsx:324-327`, lookup bez `dayId`) trening znajduje — niespójność między kartami.

### 4. (d) `nextExerciseIdForDay` przy mieszanych formatach (`plan-cycle-utils.ts:82-95`)

- Prefiks id ćwiczeń pochodzi z `day.id` (`day-N` → `ex-N`, format cyklu → `ex-<sanitized-id>`). Po podmianie id dnia stare ćwiczenia zachowują stare prefiksy, matcher nowego prefiksu ich nie łapie, licznik rusza od 1. Kolizji id NIE ma (prefiksy rozłączne), ale w jednym dniu powstaje mieszanka formatów id ćwiczeń.
- Nie wymaga fixa w X19 (twarda zasada 4: id ćwiczeń nie ruszamy). Po Z151 id dnia przestaje dryfować, więc prefiksy stabilne.
