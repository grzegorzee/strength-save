## ROOT CAUSE (zweryfikowany w kodzie 2026-07-24)

**Dwa formaty id dni:**
- Start cyklu (`startCycleWithPlan`, `completeOnboardingPlan` w `cycle-actions.ts:76,118`) woła `assignCycleDayIds(days, startDate)` (`plan-cycle-utils.ts:77-78`) → dni dostają `${startDate}-dN` i TE SAME id trafiają do planu (`savePlan(uniqueDays)`). Po starcie cyklu plan i cykl są spójne.
- Format `day-N` żyje w: domyślnym planie (`trainingPlan` w `data/trainingPlan.ts`), builderze i szablonach.

**Mechanizm rozjazdu:** `saveTrainingPlanWithRevision` (`training-plan-save.ts:67-70`) przy `syncActiveCycle !== false` woła `buildActiveCyclePlanPatch(newPlan, ...)` (`plan-cycle-utils.ts:132-140`), który podmienia `cycle.days` 1:1 na dni planu — z ich AKTUALNYMI id, bez żadnego mapowania.

**Zweryfikowane wektory dryfu (plan z `day-N` przy aktywnym cyklu z `${startDate}-dN`):**
1. `resetToDefault()` w PlanEditor → `savePlan(defaultPlan)` (id `day-N`) + domyślny sync → dni aktywnego cyklu zmieniają id na `day-N`, podczas gdy dokumenty `workouts` i drafty z tego cyklu noszą `dayId = ${startDate}-dN`.
2. `addPlanDay` (`plan-day-edit.ts:13-18,30`) ZAWSZE nadaje `day-N` (pierwszy wolny N), nawet gdy pozostałe dni mają format cyklu → dodanie dnia w PlanEditor w trakcie cyklu wstrzykuje do cyklu dzień w obcym formacie (mieszanka formatów w jednym cyklu).
3. Zapis planu z buildera/szablonu podczas aktywnego cyklu (dni szablonu mają własne id).

**Skutki (do potwierdzenia zakresu w Z150):** nawigacja Dashboardu używa `todayTraining.dayId` / `item.dayId` (`Dashboard.tsx:615,1013`); drafty treningu kluczowane `(uid, dayId, date)`; `findWorkoutForRoute` ma `allowDateFallback` (łagodzi część przypadków); prefiks id ćwiczeń zależy od formatu id dnia (`exercisePrefixForDay`, `plan-cycle-utils.ts:82-85`). Zmiana id dnia w trakcie cyklu = ryzyko: druga sesja na ten sam dzień kalendarzowy, draft nieodnajdywalny po edycji planu, mismatch prefiksu nowych id ćwiczeń.
