# PLAN X16A: progresja programowa v1 (cele tygodniowe, target vs actual, programowany deload) (2026-07-19)

> **Dla agenta wykonawcy:** WYMAGANY SUB-SKILL: `superpowers:executing-plans` (albo `superpowers:subagent-driven-development`). Kroki mają checkboxy `- [ ]`. TRYB 100% AUTONOMICZNY.
> Kontekst: wymaga WDROŻONYCH X14 i X15. Research: progresja programowa to NAJMOCNIEJ monetyzowana warstwa kategorii (Alpha Progression, RP, Juggernaut, Hevy Trainer); "a workout routine without progression is mostly useless". Nasza świadoma decyzja: silnik REGUŁOWY, nie LLM (AI nie generuje planów, decyzja z v6.10.0 obowiązuje).
> UWAGA STALENESS: plan pisany 2026-07-19 przed wdrożeniem X14/X15. Kod mógł się przesunąć: każde zadanie zaczyna się rozpoznaniem, odwołania weryfikuj przed edycją. Kandydat premium: NIE buduj gatingu w tym planie, oznacz funkcje flagą `progressionEngine` (wzorzec istniejących feature flags), gating = decyzja przy launchu monetyzacji.

**Goal:** Użytkownik z aktywnym planem widzi na start tygodnia CELE per ćwiczenie (ciężar x powtórzenia wyliczone regułowo z historii: double progression), w trakcie treningu cel serii (rozszerzenie istniejącego coacha następnej serii o horyzont tygodnia), a po tygodniu podsumowanie target vs actual. Co N tygodni (konfigurowalne, default co 5) lub po wykrytym plateau/bólu silnik proponuje TYDZIEŃ DELOAD (objętość -40%, ciężar -10%) jako jawną propozycję do akceptacji. Release train na końcu.

**Architektura podejścia:**
- **Rozszerzamy `next-set-advice.ts`, nie budujemy drugiego coacha.** Dziś: cel następnej serii z trendu + deload przy plateau. Nowe: `src/lib/progression-engine.ts` z horyzontem tygodnia: `computeWeeklyTargets(plan, history, week)` używa tych samych heurystyk (double progression: najpierw powtórzenia do górnej granicy zakresu, potem ciężar +2.5/+1 kg), per typ śledzenia z X14B (duration → cel czasu; assisted → cel mniejszej asysty).
- **Deload = propozycja, nie przymus.** Banner na start tygodnia deloadowego: "Tydzień odciążenia: proponuję -40% objętości, -10% ciężaru" z [Zastosuj] / [Pomiń]. Zastosowanie modyfikuje CELE (nie plan!): plan zostaje nietknięty, silnik po prostu liczy cele deloadowe. Zero migracji danych planu.
- **Konfiguracja:** pole `progression?: { deloadEveryWeeks?: number, enabled: boolean }` w dokumencie planu (`training_plans`), edycja w PlanEditor/PlanBuilder. Rules: dopisać do zamkniętego schematu planu.
- **Target vs actual:** czysta funkcja porównująca cele tygodnia z faktycznymi seriami; wynik na Dashboard (karta "Twój tydzień") i w podsumowaniu tygodnia AI.

**Tech stack:** React + TS, vitest (silnik = czyste funkcje, duże pokrycie), Firestore rules + `test:rules`, Playwright mock.
