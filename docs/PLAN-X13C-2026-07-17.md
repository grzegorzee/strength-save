> **Dla agenta wykonawcy:** WYMAGANY SUB-SKILL: `superpowers:executing-plans` (albo `superpowers:subagent-driven-development`). Kroki mają checkboxy `- [ ]`. Zadania PO KOLEI. TRYB 100% AUTONOMICZNY: nie pytaj, decyzje wg planu, niejasności rozstrzygaj wariantem prostszym + wpis do DECYZJE.md.
> Kontekst: wymaga WDROŻONYCH X13A i X13B. Realizuje zatwierdzoną przez usera (2026-07-17) drugą część decyzji o narzędziach naprawczych: naprawy per user wykonywane z panelu admina przez Cloud Functions (Admin SDK), z dry-run i backupem, plus dziennik wszystkich akcji administracyjnych.
> ISTNIEJE: naprawy client-side na WŁASNYM koncie (Ustawienia, od X12B za flagą isAdmin): Połącz przerwane cykle, Napraw dane historyczne, Wyczyść duplikaty treningów, Wymuś onboarding od nowa. Wzorce Functions: `functions/src/security.ts` (weryfikacja admina), `admin-api.ts`, testy vitest, `firestore-batch.ts`, `bounded-concurrency.ts`.
> Odwołania plik:linia z rozpoznania 2026-07-17. Przed edycją zweryfikuj aktualny stan pliku.

**Goal:** Admin z panelu (szczegół usera z X13B) wykonuje na DOWOLNYM koncie 4 naprawy znane z Ustawień, zawsze w trybie: dry-run (podgląd diffa) -> wykonanie z automatycznym backupem. Każda akcja administracyjna (naprawy + istniejące przełączniki uprawnień) ląduje w dzienniku `admin_audit_log` widocznym w panelu. Release train na końcu.

**Architektura podejścia:** Klient admina NIE dostaje prawa zapisu do cudzych dokumentów (rules bez zmian w write). Naprawy wykonuje callable Cloud Function z Admin SDK po weryfikacji roli admina. Algorytmy napraw są portem czystych funkcji z klienta z testami parytetu na wspólnych fixtures (żeby naprawa z panelu robiła DOKŁADNIE to samo, co naprawa z Ustawień). Bezpieczniki zamiast pytań: dry-run domyślnie, backup dokumentów do `admin_repair_backups` przed każdym apply, dziennik niezmienialny.

**Tech stack:** Cloud Functions (callable, Admin SDK), Firestore rules + testy, React + TS (panel), vitest po obu stronach, Playwright mock.

5. Parytet algorytmów: te same fixtures wejściowe dają ten sam wynik w kliencie i w Functions (testy parytetu obowiązkowe).
