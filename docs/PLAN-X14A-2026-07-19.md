> Kontekst: wynik syntezy 3 deep researchy (2026-07-19). Przypięte notatki = pain point nr 1 kategorii (migracje Hevy→Strong z tego powodu). Empty workout = drugi najczęstszy powód churnu.
> Odwołania plik:linia z rozpoznania 2026-07-19. Przed edycją zweryfikuj aktualny stan pliku.

**Goal:** (1) Użytkownik przypina do ćwiczenia trwałą notatkę (technika, ustawienia maszyny: siedzisko, pin), widoczną i edytowalną w każdej sesji z tym ćwiczeniem, niezależnie od planu. (2) Użytkownik startuje trening bez planu jednym tapnięciem ("Szybki trening"), dodaje ćwiczenia w locie, a historia/rekordy/statystyki traktują go jak każdy inny trening. Release train na końcu.

**Architektura podejścia:**
- **Klucz notatki = kanoniczna NAZWA ćwiczenia, nie exerciseId.** `exerciseId` jest niestabilne między planami (`tpl-ex-N` per instancja planu, patrz sekcja "Historia odporna na zmianę planu" w START.md). Nazwa rozwiązywana tym samym mechanizmem co historia: `resolveExerciseName` (`src/lib/exercise-name-resolver.ts`). Dla własnych ćwiczeń: nazwa z `useCustomExercises`.
- **Storage:** nowa kolekcja `exercise_notes`, dokument per (user, ćwiczenie): `{ userId, exerciseName, note, machineSettings?, updatedAt }`. Doc id deterministyczny: `${userId}_${slug(exerciseName)}` (idempotentne zapisy, brak duplikatów). Rules: zamknięty schemat + izolacja per user, jak pozostałe kolekcje.
- **Empty workout:** syntetyczny dzień `adhoc-<YYYY-MM-DD>-<ts>` ze snapshotem `dayName/dayFocus`. Dzięki architekturze snapshot+resolver trening ad-hoc NIE potrzebuje wpisu w planie, historia działa od razu. Draft przez istniejący `workout-draft-db.ts`.

**Tech stack:** React + TS, Firestore (rules + `npm run test:rules`, JDK21), vitest, Playwright mock (`npm run e2e:mock`).
