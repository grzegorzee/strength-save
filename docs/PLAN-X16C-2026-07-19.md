# PLAN X16C: aplikacja Garmin Connect IQ v1. Logowanie serii z zegarka Garmin + natywny zapis do Garmin Connect (2026-07-19, wersja 2)

> **Dla agenta wykonawcy:** WYMAGANY SUB-SKILL: `superpowers:executing-plans` (albo `superpowers:subagent-driven-development`). Kroki mają checkboxy `- [ ]`. TRYB 100% AUTONOMICZNY.
> **WERSJA 2 (decyzja usera 2026-07-19 wieczór):** zastępuje pierwotny plan X16C oparty o Garmin Health/Activity API. Powód: Health API wymaga akceptacji Garmin Connect Developer Program (gatekeeper, tygodnie, możliwa odmowa), a import cardio od userów Garmina już w większości pokrywa istniejąca Strava (auto-sync Garmin→Strava). Aplikacja Connect IQ nie ma gatekeepera (SDK darmowe, dystrybucja przez Connect IQ Store ze zwykłą recenzją) i daje więcej: logowanie serii z nadgarstka na Garminie (nie ma tego NIKT w kategorii: Strong/Hevy tylko Apple Watch/Wear OS) + trening siłowy natywnie w Garmin Connect (sesja FIT z HR, bez żadnego API). Health API import = backlog (opcja, gdyby Garmin kiedyś przyznał dostęp).
> Kontekst: wymaga WDROŻONYCH X16A (cele tygodniowe w kontrakcie dnia) i X16B (wspólny kontrakt `watch-contract.ts`, który tu REUŻYWAMY). Research: prośba o Connect IQ z r/Hevy z dopiskiem "I'd even be willing to pay for it".
> **BLOKADY ZEWNĘTRZNE tego planu:** (a) recenzja Connect IQ Store (dni), (b) test na fizycznym zegarku Garmin = KROK USERA (symulator CIQ pokrywa resztę). Obsługa wg pętli z PROMPT-WDROZENIE: wykonaj wszystko co możliwe, odłóż resztę z listą kroków usera.
> UWAGA STALENESS: przed implementacją zweryfikuj aktualne wersje Connect IQ SDK i listę urządzeń (web search).

**Goal:** Użytkownik z zegarkiem Garmin paruje go z kontem (6-cyfrowy kod), a potem na zegarku: widzi dzisiejszy trening (ćwiczenia, cele serii, przypięte notatki), odhacza serie z korektą ciężaru, dostaje rest timer z wibracją, a zegarek równolegle nagrywa natywną sesję siłową (HR, kalorie) do Garmin Connect. Ukończony trening ląduje w Strength Save jako normalna sesja (historia, rekordy, obciążenie hybrydowe). Działa bez telefonu w zasięgu (kolejka na zegarku, wysyłka po odzyskaniu łączności). Release train na końcu.

**Scope v1 (ŚWIADOMIE WĄSKI, lustro Apple Watch v1 z X16B):**
- JEST: parowanie kodem, pobranie dnia, lista ćwiczeń, cel serii, odhaczenie ze stepperem ciężaru (+/- 2.5 kg) i powtórzeń (+/- 1), rest timer z wibracją, podgląd przypiętej notatki, zakończenie treningu, natywna sesja FIT (strength) do Garmin Connect, kolejka offline.
- NIE MA w v1: edycji planu, dodawania ćwiczeń z zegarka, historii na zegarku, live-syncu z draftem na telefonie (trening z Garmina = standalone sesja zapisywana przez backend po zakończeniu; jednoczesny trening na telefonie i Garminie to nie jest wspierany scenariusz v1, guard opisany w Z125).

**Architektura podejścia:**
- **Zegarek rozmawia z NASZYM backendem, nie z telefonem.** `Communications.makeWebRequest` (przez Garmin Connect Mobile na telefonie albo Wi-Fi/LTE zegarka) na trzy endpointy Cloud Functions (HTTP): `garminPair`, `garminDay`, `garminIngest`.
- **Parowanie:** user w Ustawieniach Strength Save generuje 6-cyfrowy kod (kolekcja `device_pair_codes`: hash kodu, uid, TTL 10 min, jednorazowy). Zegarek wysyła kod → `garminPair` wydaje token urządzenia (random 32B; w Firestore `device_tokens` tylko hash + uid + label + createdAt/lastUsedAt; token w `Application.Storage` zegarka). Unlink w Ustawieniach = revoke tokenu.
- **Kontrakt danych = REUŻYCIE X16B:** `buildWatchDayContext` (kompaktowy JSON dnia: ćwiczenia, serie z celami, notatki) serwowany przez `garminDay`; zdarzenia odhaczeń idempotentne po `eventId` jak w `applyWatchEvent`. Jedna definicja kontraktu w `src/lib/watch-contract.ts`, functions importują typy/walidację z współdzielonego modułu (wzorzec testów parytetu klient<->functions z X13C).
- **Zapis treningu:** `garminIngest` przyjmuje paczkę zdarzeń + zakończenie, waliduje token, składa `WorkoutSession` (snapshot nazw ćwiczeń: architektura snapshot+resolver sprawia, że sesja z zegarka nie potrzebuje planu po stronie klienta) i zapisuje przez Admin SDK. Idempotencja: docId `garmin-<deviceId>-<workoutId>`.
- **Apka CIQ:** device app w Monkey C (Connect IQ SDK, VS Code extension, symulator), min API dobrane tak, by pokryć popularne zegarki 2022+ (Fenix 7/8, Epix 2, Forerunner 255/265/955/965, Venu 2/3, vívoactive 5; finalna lista w Z126.0). Sesja `ActivityRecording` typu strength (HR z nadgarstka) zapisywana lokalnie → Garmin sync robi resztę.

**Tech stack:** Monkey C (Connect IQ SDK 7/8), Cloud Functions (Node, wzorce auth z registration.ts), Firestore rules + `test:rules` (JDK21), vitest (kontrakt i walidacje), symulator CIQ.

## 0. TWARDE ZASADY

1. Deploy wyłącznie w FAZIE 3 (release train).
2. Bezpieczeństwo: endpointy HTTP wyłącznie po tokenie urządzenia (hash w Firestore, porównanie timing-safe jak webhook Stravy), rate limiting per token, CORS zamknięty. Kod parowania jednorazowy z TTL. Zero sekretów w apce CIQ.
3. Koszty Firestore/Functions: `garminDay` max 1 odczyt kontekstu na start treningu + odświeżenie na żądanie (bez pollingu), `garminIngest` batched (flush zdarzeń przy odhaczeniu gdy łączność, inaczej kolejka; zapis dokumentu treningu raz, na zakończenie).
4. Jednoczesny trening telefon+Garmin: `garminIngest` przy zapisie sprawdza, czy na ten dzień istnieje już completed sesja z tego samego dnia planu; jeśli tak, zapisuje jako osobną sesję ad-hoc z dopiskiem źródła (żadnego mergowania w v1, zero ryzyka nadpisania danych z telefonu).
5. Zdarzenia idempotentne (eventId), deduplikacja po stronie functions z testem.
6. i18n apki CIQ: PL+EN w resources. TDD dla całej logiki TS/functions, chirurgiczne commity, pełne bramki, DECYZJE.md po fazach.
