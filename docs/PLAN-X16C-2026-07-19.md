**Goal:** Użytkownik łączy konto Garmin w Ustawieniach (OAuth, wzorzec istniejącej Stravy). Aktywności z zegarka Garmin (biegi, jazdy, spacery) wpadają automatycznie do zunifikowanego strumienia aktywności i liczą się do obciążenia hybrydowego. Opcjonalnie (jeśli API na to sensownie pozwala): ukończone treningi siłowe eksportują się do Garmin Connect. Release train na końcu.

**Architektura podejścia:**
- **Wzorzec = istniejąca Strava.** OAuth i wymiana tokenów przez Cloud Functions (jak `functions/src` dla Stravy: client secret na serwerze, nigdy w kliencie). Nowa kolekcja `garmin_activities` o kształcie zgodnym z `UnifiedActivity` (source: 'garmin'); `useActivities` dostaje trzecie źródło.
- **Model danych Garmin:** Garmin Activity API działa w modelu push (webhook/ping do naszego endpointu po nowej aktywności), nie pull jak Strava. Endpoint = Cloud Function HTTP z weryfikacją; mapowanie payloadu na nasz typ w czystej funkcji (testy na payloadach fixture z dokumentacji).
- **Deduplikacja źródeł:** user może mieć Garmin → Strava auto-sync i naszą Stravę jednocześnie: ta sama aktywność przyszłaby dwa razy. Reguła dedupe: ta sama data+czas startu (+/- 2 min) + typ + dystans (+/- 2%) → pokaż raz (priorytet: strava > garmin, stała jawna w kodzie z testem).
- **Eksport siły (Z126):** TYLKO jeśli aktualne API umożliwia zapis aktywności typu strength z zewnątrz. Research na początku zadania; jeśli nie → zadanie oznaczone NIEWYKONALNE ZEWNĘTRZNIE z uzasadnieniem w DECYZJE.md (to jest poprawny wynik, nie porażka).

**Tech stack:** Cloud Functions (Node, wzorce Strava OAuth + scheduled), Firestore rules + `test:rules`, React + TS, vitest, fixtures payloadów.
