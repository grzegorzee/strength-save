## Wszystkie obszary (szczegóły)

### Reguły Firestore

Reguły są w dobrym stanie: per-user isolation działa (userId sprawdzany na create/update/delete, cross-user read zablokowany), eskalacja uprawnień jest zablokowana (role/status/access w blockliście, create users tylko przez Cloud Functions), kolekcje wrażliwe (strava_connections, api_keys, audit logs) są zamknięte na klienta, default deny obowiązuje, a znany gotcha resource==null jest świadomie obsłużony i pokryty testami emulatora (scripts/test-firestore-rules.mjs, 30+ przypadków). Główne słabości to deny-lista zamiast allow-listy przy update profilu (przyszłe pola serwerowe będą domyślnie edytowalne przez usera, a email/uid można już dziś spoofować w panelu admina) oraz existence oracle na deterministycznych ID workoutów.

- **[medium] Update profilu users oparty na deny-liście: user może nadpisać email/uid, a każde nowe pole serwerowe jest domyślnie edytowalne** (`firestore.rules:33-53`)
  - Wpływ: User może zmienić pole email/uid we własnym dokumencie i podszyć się pod inny adres w panelu admina (desynchronizacja z Firebase Auth, mylące filtrowanie/wyszukiwanie userów przez admina). Groźniejszy jest wzorzec: każde NOWE pole kontrolowane przez serwer (np. przyszły tier, limity AI) będzie user-writable, dopóki ktoś nie pamięta o dopisaniu go do blocklisty.
  - Naprawa: Odwrócić logikę na allow-listę: allow update tylko gdy affectedKeys().hasOnly(['displayName','photoURL','notificationPrefs','fcmTokens','onboarding','onboardingCompleted','units','language',...]). Dodać test reguł: update pola 'email' przez usera = deny.
- **[low] Existence oracle: resource==null + deterministyczne ID workoutów pozwala sprawdzić czy inny user trenował danego dnia** (`firestore.rules:66-67`)
  - Wpływ: Dowolny zalogowany user (nawet pending_verification lub z access.enabled=false) znający uid ofiary rozróżnia: getDoc nieistniejącego = sukces (pusty snapshot), istniejącego cudzego = PERMISSION_DENIED. Czyli może enumerować po datach i dayId, kiedy ofiara trenowała. Treść nie wycieka, tylko fakt istnienia. To samo dotyczy plan_cycles (firestore.rules:139), ale tam ID nie są przewidywalne.
  - Naprawa: Zawęzić gałąź resource==null do własnych dokumentów: `(resource == null && workoutId.matches('workout-' + request.auth.uid + '-.*'))` — regresja pierwszego treningu dalej przechodzi (klient pyta tylko o własne ID), a oracle znika. Dodać test: getDoc nieistniejącego workouta z cudzym uid w ID = deny.
- **[low] Brak gałęzi resource==null w read dla measurements, weekly_summaries, chat_messages, strava_activities, ai_usage** (`firestore.rules:80-81`)
  - Wpływ: Dziś klient używa wyłącznie query z where('userId','==',uid) (src/hooks/useFirebaseWorkouts.ts:94-96, src/hooks/useWeeklySummary.ts:21-22), więc nic nie pada. Ale każdy przyszły getDoc po ID nieistniejącego dokumentu (np. measurement-{timestamp}) dostanie PERMISSION_DENIED zamiast pustego snapshotu. Ten dokładnie gotcha już raz zablokował rozpoczęcie pierwszego treningu (komentarz w firestore.rules:63-65).
  - Naprawa: Świadoma decyzja do udokumentowania: albo dodać `resource == null` (zawężone do własnego prefiksu ID jak w findingu wyżej), albo zostawić i pilnować w code review, by klient nigdy nie robił getDoc po deterministycznym ID w tych kolekcjach.
- **[low] chat_messages: martwa powierzchnia zapisu po usunięciu AI Chat w v6.7.0** (`firestore.rules:125-134`)
  - Wpływ: Klient nie pisze już do tej kolekcji, ale reguła wciąż pozwala każdemu aktywnemu userowi tworzyć nieograniczoną liczbę dokumentów chat_messages (śmieci w bazie, koszty storage, szum w danych). Niewykorzystywana powierzchnia zapisu to klasyczny dług bezpieczeństwa.
  - Naprawa: Skoro feature usunięty: zamienić na `allow read, write: if false;` (lub zostawić read dla admina do wglądu w stare dane) i zaktualizować test reguł.
- **[low] Brak walidacji schematu i rozmiaru pól przy create/update danych treningowych** (`firestore.rules:68-71`)
  - Wpływ: User może zapisać do workouts/plan_cycles/training_plans dowolne pola i dokumenty do 1 MB. Dane czyta panel admina i sync; zniekształcony dokument (np. date jako number, exercises jako string) może wywalić widoki lub naliczanie statystyk u samego usera, a masowy zapis dużych dokumentów podbija koszty odczytu admina. Dotyczy tylko własnych danych, więc edge case.
  - Naprawa: Minimalna walidacja na create/update: `request.resource.data.date is string && request.resource.data.exercises is list && request.resource.data.keys().hasOnly([...])` dla workouts; analogicznie dla plan_cycles i training_plans (durationWeeks is int, days is list).

Pomysły UX z tego obszaru:
- Po stronie klienta mapować PERMISSION_DENIED z reguł na czytelny komunikat (np. konto zawieszone / oczekuje weryfikacji) zamiast generycznego błędu Firestore, bo hasSelfAccess odrzuca też userów pending_verification i access.enabled=false.

### Autentykacja

Fundamenty są solidne: gate weryfikacji email jest wymuszany serwerowo (firestore.rules wymaga status 'active' w hasSelfAccess, callables sprawdzają hasCallableAppAccess), kody weryfikacyjne mają cooldown 60 s, TTL 10 min, limit 5 prób i hash SHA-256 z pepperem. ### Zapis treningu (główna ścieżka krytyczna): draft IndexedDB → checkpoint/final sync do Firestore

### OFFLINE + KOLEJKA SYNC

Architektura offline jest solidna w warstwie przechowywania: draft treningu żyje w IndexedDB (workout-draft-db), kolejka nieudanych final-synców w localStorage (workout-sync-queue), a Firestore ma persistentLocalCache. Zabicie apki z niepustą kolejką nie gubi danych. Słabe punkty to odtwarzanie: kolejka nie synchronizuje się automatycznie po powrocie online (poza otwartą stroną WorkoutDay), wpisy kolejki tracą przy odczycie pola cloudUpdatedAt/exerciseNames/dayName (deserializacja je wycina), konflikt dwóch urządzeń offline kończy się cichym nadpisaniem całego treningu (last-write-wins bez merge), a licznik pendingOps podwójnie liczy tę samą sesję. Duplikatów sesji przy retry nie znalazłem (deterministyczne ID sesji + transakcja chronią przed nimi).

### Stabilność + obsługa błędów

Fundamenty są solidne: registerType 'prompt' + skipWaiting:false (update PWA nie przerywa treningu), guard blokady update'u podczas aktywnej sesji, lazyWithRetry z jednorazowym reloadem na chunk error, timery w komponentach mają poprawne cleanupy, a onSnapshot w hookach jest odsubskrybowywany. Telemetria nie zawiera PII (tylko userId + liczniki zdarzeń, rules ograniczają odczyt do self/admin). Główne ryzyko: flaga blokady update'u w localStorage nigdy nie wygasa i zostaje na stałe po zabiciu apki w trakcie treningu, co po deployu prowadzi do pełnoekranowego ErrorBoundary przy nawigacji (chunk error bez auto-reloadu). Do tego ErrorBoundary jest tylko jeden (root) i nigdzie nie raportuje crashy.
